## Phase 3, part 1 ‚Äî live price calculator + shared evaluator (built + verified 2026-07-19)

The first user-visible slice of Phase 3: the public **real-time price calculator**, replacing the Phase 2 static/disabled preview. Deliberately bounded to **the shared pricing evaluator + the public calculator**, leaving the admin **visual Calculator Field/Formula Builder** (the non-technical authoring UI, TECHSPEC ¬ß6.4) to **Phase 3 part 2** ‚Äî the same "prove the risky shared foundation first, then the authoring UI" split used for Phase 2. Rationale: the novel engineering (a safe evaluator shared by client + server, the ¬ß7 edge cases, the real-time UX) all lives in the evaluator + public component; once proven against hand-authored `formula` JSON, the visual builder is a self-contained admin component that emits that same JSON. **Migration-free** ‚Äî the `calculatorFields` array + `formula` JSON have existed since Phase 1.

### The engine decision (homework ‚Äî worth a fresh session knowing)

TECHSPEC ¬ß5/¬ß6.3 named `json-logic-js` "(or equivalent safe evaluator)". Re-researched at implementation time (the standing rule that "current best practices" are re-checked, not trusted from when the plan was written): **`json-logic-js` is effectively unmaintained** (no npm release in 12+ months) ‚Äî a liability against the "stay-current / periodic-maintenance-only" goal. Its maintained successor is **`json-logic-engine`** (MIT, TS-native, ESM+CJS, ~5√ó faster, v5.0.7). **Decision: keep JSONLogic as the stored/interchange format** (portable, library-agnostic, exactly what the part-2 visual builder will emit) **but evaluate it with a ~50-line zero-dependency in-house arithmetic-subset interpreter** rather than adding either library. Why: the pricing domain is tiny and closed (`var`, `+ ‚àí √ó √∑`, `min`, `max`); owning the evaluator gives full, unit-tested control over the money-math edge cases that actually matter (division by zero, NaN/Infinity, the ¬ß7 non-positive-total rule), adds **zero supply-chain surface** ‚Äî keeping the strict `npm audit --audit-level=high` CI gate clean (recall the `tmp`/lhci episode, where one added dev tool breached that gate) ‚Äî and involves **no lock-in**: because the on-disk format is standard JSONLogic, swapping in `json-logic-engine` later (if operator needs ever outgrow arithmetic) is a drop-in with **no data migration**. This is the same "custom because the domain is small, well-understood, and we want control" reasoning used for the TOTP crypto.

### What was built (files in this slice)

### Decisions worth a fresh session knowing

- **Two price paths, mutually exclusive for the total.** Default (no formula) = Œ£ signed `unitPrice √ó value` ‚Äî covers the common service with no formula to author, and matches the field-level model in FUNCTIONALITY ¬ß5.3. A custom JSONLogic `formula`, when present, is authoritative and evaluated over **raw field values** (fixed costs, %-adjustments, groupings live there). The breakdown shows per-field contributions in the default path (they sum to the total) and entered values under a formula ("where applicable", ¬ß4).
- **Required-field gating over always-live.** An "estimate" that silently treats unfilled required inputs as 0 invites false trust; gating on **required only** (not all fields) keeps it responsive (optional fields still default to 0 and update instantly). Hard, blocking validation (format errors, etc.) belongs at PDF-generation time (Phase 4). **0 is a value, blank is not.**
- **The evaluator is pure and framework-free**, so it's the one piece fully unit-testable in-sandbox (Node `--experimental-strip-types`) and reusable verbatim by the Phase 4 PDF and any server-side validation ‚Äî the on-screen price can never drift from the quoted price (TECHSPEC ¬ß3/¬ß6.3).
- **Currency is locale-formatted EUR** (`‚Ç¨1,234.50` in EN, `1.234,50 ‚Ç¨` in FR/DE), not a hardcoded symbol ‚Äî correct per-locale placement for the trilingual site.

### Issues found during testing (and fixed) ‚Äî worth a fresh session knowing

1. **One `tsc` error (test-only).** A JSONLogic rule literal that mixed two differently-keyed objects in one array (`{'+':‚Ä¶}` and `{var:'c'}`) was inferred as a union with phantom `undefined` members that won't widen to `JsonLogic`; fixed by annotating the const `: JsonLogic` (contextual typing then checks the literal directly). **Lesson:** a structured-AST literal sometimes needs an explicit type annotation to widen to a recursive union ‚Äî the same class of thing caught only by `tsc`, not the build.
2. **The toggle didn't respond to clicks.** The switch's visible slider sat inside a `<span>`, so clicking it never toggled the 0√ó0 hidden checkbox ‚Äî only the far-left field-name label (which carries `htmlFor`) did. Fixed by making the slider wrapper a `<label>` (implicit association) and adding a `:focus-visible` ring. **Lesson:** a CSS switch built on a visually-hidden checkbox must be wrapped in a `<label>`, or the visible control is dead to the mouse.
3. **The total showed before required fields were filled** ‚Äî and (Filip's point) a `0` shouldn't read as "empty". Reworked to gate the total on required-field completion, with `0` counting as filled (see the decision above). Good general reminder: for a live calculator, distinguish "not yet entered" (blank) from "entered zero".
4. **The estimate disclaimer showed on no-calculator services**, where there's no estimate to disclaim. Gated the before-disclaimer on `hasCalculator`.
5. **Pre-existing (not this slice) ‚Äî admin/TOTP e2e cold-start flakiness.** One `test:e2e` run showed `admin.e2e`/`totp.e2e` `beforeAll` timeouts (30 s) while the first `/admin` hit compiled the admin bundle **and** ran Payload's Drizzle schema push; the public suites (incl. the new calculator test) passed. ### Decisions worth a fresh session knowing

- **ID-based service URLs, slugs deferred.** `Services` has no slug field; adding one is a schema change (migration), and this slice was deliberately kept migration-free (lower risk, and the sandbox can't run `migrate:create` without the embedded-Postgres workaround). `/services/[id]` works now; slugs land in Part 2 with the Projects snapshot migration. Staging isn't indexed, so the URL change later is safe.
- **`overrideAccess: false` everywhere in the data layer**, not a hand-written `where: { _status: 'published' }`. Reuses the exact access policy `rest.int.spec.ts` already proves, so drafts can't leak through a second, drifting code path.

## Phase 1, part 1 ‚Äî TOTP 2FA (built 2026-07-06)

Payload has no native 2FA, so this is custom code layered on top of Payload's own email/password auth, per `docs/TECHSPEC.md` ¬ß6.6. `docs/FUNCTIONALITY.md` ¬ß5.1 treats 2FA as **mandatory, not optional**.

### What was built (files in this slice)

- `src/lib/totp/` ‚Äî the crypto/OTP toolkit, all Node-runtime:
  - `keys.ts` ‚Äî one root secret (`TOTP_ENCRYPTION_KEY`, separate from `PAYLOAD_SECRET` for key separation) ‚Üí two HKDF-derived subkeys (one for AES-256-GCM encryption, one for HMAC signing).
  - `crypto.ts` ‚Äî AES-256-GCM encrypt/decrypt of the TOTP secret at rest (`iv:authTag:ciphertext`, base64).
  - `otp.ts` ‚Äî `otplib` wrappers (SHA-1/6-digit/30s defaults, matching mainstream authenticator apps); `epochTolerance: 30` (¬±1 window) and `afterTimeStep` replay protection.
  - `stepUpToken.ts` ‚Äî plain HMAC-SHA256 "second-factor-verified" token (`uid`, `exp`), 2h TTL matching Payload's session; `timingSafeEqual` compare.
  - `requestHelpers.ts` ‚Äî parse/verify the step-up cookie from headers or Payload's cookie map; build set/clear `Set-Cookie` values.
  - `rateLimit.ts` ‚Äî Upstash Redis sliding-window limiter (5/5min) when `UPSTASH_REDIS_REST_URL/TOKEN` are set; an in-memory fallback for local/CI (same "real service in prod, local stand-in for dev/CI" pattern as S3Mock; not safe multi-instance, warns if it ever activates in prod).
  - `qr.ts` ‚Äî QR data-URL for the enrollment screen.
- `src/collections/Users.endpoints.ts` ‚Äî custom endpoints under `/api/users/totp/*`: `setup` (mint secret + QR; re-enroll requires current step-up), `enable` (confirm a code, flip `totpEnabled`, set step-up cookie), `verify` (per-login second factor, sets step-up cookie, user+IP rate-limited), `disable` (requires step-up **and** password re-check), `status`.
- `src/access/requireTotpVerified.ts` ‚Äî access wrapper: denies unless there's a `req.user`, `totpEnabled`, **and** a valid step-up cookie for that user id; then delegates to the wrapped access fn. **This is the security boundary.**
- `src/proxy.ts` ‚Äî Next.js 16 Node-runtime request interceptor (the `proxy.ts` convention that replaced `middleware.ts` in Next 16) gating `/admin/:path*`: a password-only session hitting a protected admin route is redirected to `/admin/totp-verify`. UX/defense-in-depth; verifies the `payload-token` JWT itself (see the key-derivation bug below).

### Two real bugs surfaced during testing (both worth a fresh session knowing)

1. **Proxy verified the session JWT against the wrong key ‚Äî the headline bug.** `src/proxy.ts` verified the `payload-token` with the **raw** `PAYLOAD_SECRET`. Payload does **not** sign with the raw value: on init it derives `crypto.createHash('sha256').update(config.secret).digest('hex').slice(0, 32)` (see `node_modules/payload/dist/index.js`, `BasePayload` init) and signs/verifies with `TextEncoder().encode(...)` of **that**. So every real token failed with `signature verification failed`; `verifiedPayloadUserId` returned `undefined`; the proxy fell through to `NextResponse.next()` and never redirected ‚Äî the symptom was a password-only session landing on `/admin/collections/users` and seeing an empty "Nothing found" list (the access wrapper correctly denied the data; only the redirect UX was missing). The integration test masked it because it hand-signed its test tokens with the raw secret too ‚Äî test and buggy proxy agreed with each other while both disagreed with reality. **Fix:** `payloadJwtKey()` in `proxy.ts` reproduces Payload's exact derivation and is **exported and reused by the int test**, so the two can't drift again. **Lesson:** the one piece of custom code re-deriving Payload's own behavior is exactly where reality diverged ‚Äî from a confidently-worded but unverified comment ("does not hash or transform"). Verify such assumptions against the installed source, and never let a test re-implement the very thing it's meant to check.
   - **Diagnostic red herring worth not repeating:** this was first mis-attributed to Turbopack "not running" the proxy, because `.next/.../middleware-manifest.json` was empty. That file is empty *by design* for a **Node-runtime** proxy ‚Äî those register in `functions-config-manifest.json`, not `middleware-manifest.json`. A brief switch of `dev` to `--webpack` was a wrong turn (it only added slow-compile timeouts) and was reverted; `dev`/`devsafe` stay on Turbopack. What actually pinned it was temporary `console.warn` instrumentation in the proxy: the dev-server log showed `jwtVerify FAILED: signature verification failed`, which is unambiguous. When a Node-runtime proxy "doesn't fire," instrument it and read the server console rather than reasoning about manifests.
2. **TOTP replay-window flake in the e2e suite (a test bug, not a code bug).** The "new session" test reused the account the enrollment test had *just* enrolled. When its login code fell in the same 30-second TOTP window whose time-step enrollment had already consumed, otplib's replay protection (correctly) rejected it ‚Äî a ~50‚Äì70%-of-runs timeout depending on where the window boundary fell. The 2FA logic is right; the test was unrealistic (real "new session" logins happen long after enrollment). **Fix:** `seedEnrolledTestUser` seeds a dedicated, already-enrolled account with an untouched time-step; the new-session test uses that, which also removes a hidden test-1‚Üítest-2 ordering dependency.

- `npm run test:e2e`: **now green in CI.** The `deploy-staging` job that shipped Phase 1 `needs: verify`, which runs the full e2e suite ‚Äî so all 6 passed there (the replay-flake fix included), plus Filip's earlier manual redirect check. `npm run devsafe` (clean `.next` + restart) is required after any `proxy.ts` change ‚Äî Next never hot-reloads the proxy file.

### Decisions worth a fresh session knowing

- **Push in dev/CI, migrate on deployed stages** ‚Äî Payload's own recommended split, kept (not "push:false everywhere"). CI's `verify` job keeps pushing on boot against its ephemeral Postgres, so the test suite is untouched; only real stages are migrations-only. Having committed migrations present does **not** affect push mode (push ignores the migration files), so `test:int`/`test:e2e` stay green.
- **Migrate in CI, not `prodMigrations`** ‚Äî the serverless-correct choice (see above), and what Payload's docs recommend for anything that isn't a long-running container.
- **Neon: migrations use the DIRECT (unpooled) URL** ‚Äî DDL breaks through Neon's PgBouncer pooler (transaction pooling). Runtime stays pooled; only the CI migrate step uses the direct URL. - **`TotpEncryptionKey` required, Upstash optional** ‚Äî encodes the actual severity: no TOTP key = broken admin (hard-fail the deploy); no Upstash = degraded-but-working rate limiting (tolerated pre-launch).

### Sandbox constraints hit (and how they were worked around)

- **The `payload` CLI can't load the TS config in-sandbox** ‚Äî it uses `tsx`‚Üíesbuild, and `@esbuild/linux-x64` isn't in the Windows-mounted `node_modules` (the same "native binaries are Windows-only" quirk from Phase 0). Worked around exactly as the content-model session did: load the config with Node 22's native `--experimental-strip-types` + a small `@/`-alias/extensionless resolve hook, and call `createMigration`/`migrate` on `payload.db` directly. One wrinkle: Payload's generated migration imports `MigrateUpArgs`/`MigrateDownArgs` as *values* (`import { MigrateUpArgs, MigrateDownArgs, sql }`), which `--strip-types` can't elide (only real `tsx`/esbuild does), so the sandbox apply-test patched that one import line in a scratch copy; **the committed file keeps Payload's original import** (correct for CI/Filip's toolchain).
- **`bwrap --unshare-pid --die-with-parent`** ‚Äî every bash call is its own PID namespace, so **no process survives between calls** (background/`setsid`/`nohup` all die with the call). But the on-disk `pgdata` persists, so the pattern is: warm the cluster (`initdb`) once, then reuse the data dir in later calls (start is ~0.4s, no initdb). Also note `pkill -f postgres` **self-matches** the calling shell (its command line contains "postgres") and kills the call ‚Äî use `pkill -x postgres` or, since nothing survives a call anyway, don't bother.
- **Stale mount reads** ‚Äî after editing `package.json` via the Edit tool, the Linux side served a truncated pre-edit copy (2473 bytes, unterminated mid-string), which broke Node's module resolution for any in-sandbox eval. The Read tool (authoritative Windows view) confirmed the file was correct; the adapter-options check was rerun mount-independently (importing `@payloadcms/db-postgres` by absolute path). Same Windows‚ÜîLinux flakiness as always ‚Äî trust Read/Edit/Write.

## How Phase 0 went (the Lambda spike)

Scaffolding (Next.js 16.2 + Payload 3.85 + Postgres/Drizzle + SST v4/OpenNext + GitHub Actions) went roughly as planned; getting Payload to actually serve on Lambda took five real, non-obvious fixes, none of which invalidated the architecture:
1. OpenNext copies externalized packages (`pg`, `pino`) into the Lambda bundle via symlinks that Windows can't reliably produce as Linux symlinks ‚Üí **deploys go through CI (Ubuntu) only, never native Windows.**
2. `sharp` is excluded from the main server bundle by OpenNext and ships arch-specific binaries that don't cross-compile (x64 CI ‚Üí arm64 Lambda) ‚Üí omitted entirely (nothing in Media uses it; it's now also removed from `package.json`, keeping deps honest).
3. Media uploads need an explicit storage adapter (`@payloadcms/storage-s3`) ‚Äî SST linking a bucket only wires IAM, it doesn't make Payload use it.
4. `admin/importMap.js` is generated, not static ‚Äî regenerate (`npm run generate:importmap`) after any collection/field/plugin that contributes admin UI; silently stale otherwise.
5. IAM's `aws:RequestedRegion` doesn't scope global services (CloudFront/IAM/Route 53) ‚Üí CloudFront needed its own unconditioned statement.

## Environment quirks worth knowing (specific to this Cowork/Claude sandbox, not the app)

- The Linux sandbox's view of the Windows-mounted repo has intermittent caching problems: `git status`/`git diff`/`cat` sometimes show stale or truncated content. **Trust the Windows-native `Read`/`Edit`/`Write` tools over bash; cross-verify with `git show HEAD:<path>`.** When bash's copy of a file looks stale, rewrite it via a heredoc (`cat > path <<'EOF' ... EOF`) from content confirmed by `Read`.
- **Do not run `git add`/`commit`/`push` from the sandbox** ‚Äî a stale `.git/index.lock` once could not be removed (`rm` returns `Operation not permitted` on the Windows mount). Filip commits via his own tooling (GitHub Desktop). The same permission quirk means the sandbox **cannot delete files** on the mount ‚Äî e.g. `smoke-test-tmp.mjs` (a harmless leftover, untracked) had to be left for Filip to delete manually.
- `npm install`/`sst install`/large builds can exceed a single sandbox command's time limit; retrying converges.
- **Turbopack cannot run in this sandbox** ‚Äî only WebAssembly SWC bindings are available, and Turbopack requires native bindings (`Turbopack is not supported on this platform`). For any in-sandbox `next build`/`next dev` reproduction, use `--webpack`. Filip's real machine (Windows) uses Turbopack, which is the Next 16 default for both `dev` and `build`; keep that distinction in mind when reproducing bundler-adjacent behavior (it's why the Turbopack red herring above was hard to rule out from the sandbox alone).
