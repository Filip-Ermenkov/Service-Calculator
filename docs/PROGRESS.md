## CI content seeding + `/services/[slug]` audit coverage (built + verified 2026-07-21)

The last **fully in-repo, unblocked** launch-readiness gap, carried across several prior slices as "add `/services/[slug]` to the axe gate once CI can seed a published service." Chosen as this slice because every remaining *feature* is externally gated (Phase 4 part 2 needs the `bulbau.lu` domain + a verified SES identity; Phase 5 needs a DeepL key) — whereas this closes a real, repeatedly-flagged coverage hole using only code, and is **fully CI-provable**. **No new dependencies, no schema change, no migration — `git revert`-able.**

### The gap it closes

CI's `verify` job runs against an **empty ephemeral Postgres**, so `/services/[slug]` — the page the **live calculator and the Download-PDF button** live on, i.e. the core interactive surface — always 404-ed there. Consequences, all now fixed: it was **excluded from the axe gate** (`tests/e2e/accessibility.e2e.spec.ts`) and the **Lighthouse gate** (`lighthouserc.cjs`), and the calculator + `/api/quote` **e2e tests could only `test.skip()`**. So the most interaction-heavy, most a11y-risky public page had *zero* automated audit coverage.

### The homework (re-checked at implementation time, per the standing rule)

Payload 3.x's blessed way to run a standalone Local-API script is **`payload run <file>`** (docs: "Using Payload outside Next.js") — it loads env the way Next.js does and initializes tsx, so no `dotenv` / extra transpiler is needed. The seed uses `getPayload({ config })` (the same entry the existing `tests/helpers/seedUser.ts` uses), `context: { disableRevalidate: true }` to skip the ISR hooks (which throw outside a request scope — `src/lib/revalidate.ts`), and is idempotent via delete-then-create on a fixed slug.

### What was built (files in this slice)

- **`tests/helpers/sampleContent.ts` (new, dependency-free)** — the canonical constants (`SAMPLE_SERVICE_SLUG = 'ci-sample-service'`, `SAMPLE_SERVICE_PATH_EN`). Deliberately imports **nothing** from `payload`, so Playwright specs can import the slug without dragging the whole config into the test process.
- **`tests/helpers/seedContent.ts` (new)** — `seedSampleContent()`: idempotently publishes ONE service with the fixed slug and one of each calculator field type (number `area` required, dropdown `finish`, toggle `rush`), no `formula` (exercises the default per-field summation path), no media (renders the placeholder icon — no S3 needed). **Safety-guarded:** refuses to run unless `ALLOW_CONTENT_SEED=true` **and** `NODE_ENV !== 'production'`, so a stray `payload run` can never write to a real Neon/staging DB.
- **`tests/helpers/seedContent.run.ts` (new)** — the `payload run` entry point. **Uses a TOP-LEVEL `await`** (see the bug below) so `payload run` waits for the async seed to finish; `process.exit(0)` at the end because `getPayload` holds a pg pool open.
- **`package.json`** — a `seed:ci` script: `payload run tests/helpers/seedContent.run.ts`.
- **`.github/workflows/ci.yml`** — a `Seed sample CMS content` step in `verify`, **after `test:int`, before `build`** (so the build statically pre-renders the service page just as production would), with `ALLOW_CONTENT_SEED: 'true'`. The seeded row persists in the job's Postgres **service container** for every later step, so this single seed covers **build → Lighthouse → e2e** (they share one Postgres for the whole job).
- **`lighthouserc.cjs`** — added `http://localhost:3000/en/services/ci-sample-service` to the audited URL set (now 5 URLs × 3 runs); the `meta-description` + `font-size` hard gates now hold on the service-detail template too.
- **`tests/e2e/accessibility.e2e.spec.ts`** — a new axe test for `/en/services/ci-sample-service`; skips on a 404 (local empty DB) so it's CI-deterministic without breaking local runs.
- **`tests/e2e/frontend.e2e.spec.ts`** — the calculator + `/api/quote` tests rewritten to navigate **directly to the known seeded slug** (skip on 404) instead of the old home-card dance — deterministic in CI, still tolerant locally.
- **Two never-audited defects fixed on that page** (surfaced only because it's now under the gates): an inline **`0.7rem` (11.2px)** hint in `ServiceCalculator.tsx` → `0.75rem` (the earlier legibility slice only swept `globals.css`), and the required-`*` asterisk contrast — a higher-specificity `.calc-label span` rule was overriding `.form-required`, rendering the (aria-hidden) asterisk **grey at ~2.5:1** on white; a scoped `.calc-label .form-required` rule restores the intended **~4.9:1 brand orange** (verified with exact WCAG math).

### Decisions worth a fresh session knowing

- **One seed step, not a Playwright globalSetup.** Because the `verify` job's Postgres is a single service container for the whole job, seeding once before `build` makes the row visible to the build, Lighthouse (separate `next start` process), **and** the e2e `next dev` server. A globalSetup would only cover e2e (not the earlier Lighthouse step) and would also write to a developer's dev DB on every local `test:e2e` — deliberately avoided.
- **Guarded, opt-in seeding.** `ALLOW_CONTENT_SEED=true` is a hard opt-in (the script throws without it) + `NODE_ENV=production` refusal, so the runner is safe to keep in the repo. Locally it writes to whatever `.env` `DATABASE_URL` is set — idempotent and clearly labelled ("CI Sample Service"), deletable in one click; in CI the DB is ephemeral.
- **Default price path on purpose.** No `formula`, so the calculator exercises the Σ-signed-`unitPrice×value` path (the common case). The custom-formula path is already covered exhaustively by the pure unit tests, so re-covering it here would add nothing.
- **Skip-on-404 keeps local runs tolerant.** The new axe/e2e tests skip when the sample page 404s (a local empty DB), so they're deterministic in CI (seeded) without forcing every developer to seed first.

### The bug found during testing (and fixed) — worth a fresh session knowing

- **Floating promise vs. `payload run` → a silent no-op.** The first runner did `seedSampleContent().then(() => process.exit(0)).catch(...)` (a *floating* promise, avoided top-level await over a since-disproven tsc worry). `payload run` only waits for the module's **synchronous** top level to finish, then exits — abandoning the floating promise **before Payload even booted**. Symptom: `npm run seed:ci` printed **nothing at all** (no Payload boot logs, no `[seed]` line, no error) and wrote no data. **Fix:** a **top-level `await seedSampleContent()`** in the runner (which is exactly what the Payload docs' seed example uses; `module: esnext` supports TLA, so `tsc` is fine). **Lesson:** with `payload run` (and standalone ESM scripts generally), the entry must `await` its async work at the top level — a fire-and-forget promise is killed on module-eval completion, and the failure is invisible (no output), not a stack trace.

Same standing rule (the sandbox can't run the real toolchain; **CI / Filip's machine is the authority**). **In-sandbox:** the three new TS files + the two edited specs parse under `--experimental-strip-types`; the seed's `data` shape checked line-by-line against the generated `Service` type in `src/payload-types.ts` (field types / `sign` / `_status` / options all assignable); `ci.yml` re-parsed as valid YAML with the seed step correctly ordered after `test:int` and before `build`; `lighthouserc.cjs` re-`require`d (5 URLs, `font-size`/`meta-description` still asserted); every calculator/service-page color pair verified against the exact WCAG relative-luminance formula (all ≥4.5:1 after the asterisk fix). **On Filip's machine (2026-07-21):** after the top-level-await fix, `npm run seed:ci` prints the `[seed] published sample service` line and the seeded page renders; **all tests pass as described** (the previously-skipped axe/calculator/quote tests now run green). ## CI schema-drift guard (built + verified 2026-07-21)

The migration-safety hardening that Phase 1.5 flagged as "still not done" and "Immediate next steps" carried as item 4, finally closed. Chosen as this slice because **every remaining *feature* is externally gated** (Phase 4 part 2 needs the `bulbau.lu` domain + a verified SES identity; Phase 5 needs a DeepL key; Phase 6 needs the domain + the client's real legal details) — so a slice that can be *implemented and fully proven end-to-end* has to be an in-repo, CI-provable one, and this is the highest-value such item left: it protects the migrate→deploy pipeline, is a direct AWS Well-Architected **Operational-Excellence** hardening step, and is entirely `git revert`-able. **CI-config only — no app code, no schema, no migration, no dependency.**

### The gap it closes

The `verify` job runs with Drizzle **`push` ON** (`NODE_ENV` unset ⇒ `src/payload.config.ts`'s `push: NODE_ENV !== 'production'` is true), so it auto-syncs the schema to the config on boot and therefore **passes happily on an un-migrated schema change** — a new field/collection that nobody ran `migrate:create` for is invisible to `lint`/`typecheck`/`test:int`. Then `deploy-staging`'s `payload migrate` runs with **`push` OFF** and applies **nothing**, shipping code whose expected columns/tables were never created on staging → runtime errors or a broken deploy. Nothing in CI caught this before; the drift was only ever found by hand (Phase 2b's migration was verified drift-free manually with exactly the command this slice automates).

### The homework (re-checked at implementation time, per the standing rule)

The community-blessed pattern (payloadcms/payload discussion #10978 — the "replicate Django `makemigrations --check`" thread) is `payload migrate:create --skip-empty` + detect whether a file appeared. **Verified against the *installed* `payload@3.86.0` source, not the search results** (the standing "verify assumptions against installed source" rule earned its keep here):

- **There is NO `--check` flag in 3.86.** A web-search summary claimed one exists; `node_modules/payload/dist/bin/migrate.js` only recognizes `skipEmpty` and `forceAcceptWarning`. Using `--check` would have been a silent no-op. (An official non-interactive flag is still only a feature request — issue #14941.)
- **`migrate:create` needs no DB and causes no push.** `migrate.js` inits Payload with **`disableDBConnect: true`** (for the `migrate:create` command specifically) and `disableOnInit: true`, so the check never connects to Postgres and never pushes — it purely diffs the schema built from the config against the snapshot (`.json`) of the latest committed migration (`@payloadcms/drizzle/.../buildCreateMigration.js`).
- **`--skip-empty` behaviour, exactly:** in `buildCreateMigration.js`, when the up/down SQL is empty it `process.exit(0)`s **without** writing a file or prompting; when there ARE changes it writes `<ts>_<name>.{ts,json}` into the migration dir and logs `Migration created at …`. So no-drift = clean exit + no file; drift = a new file.
- **The one hang risk:** an *ambiguous* enum/column **rename** drops drizzle-kit into an interactive TUI ("Is X created or renamed?") that `--skip-empty` does **not** bypass (discussion #10978, mrobinsonFS Sep 2025). Neutralized with `</dev/null` (the prompt reads EOF and the command exits non-zero) + a `timeout` (belt-and-suspenders against any hang).

### What was built (this slice)

- **`.github/workflows/ci.yml`** — one new step in `verify`, **after `test:int`, before the content-seed step** (schema is in a known state; the seed/build come after and are unaffected):
  ```
  timeout 180s npm run migrate:create -- ci_drift_check --skip-empty </dev/null 2>&1 | tee drift-check.log
  status=${PIPESTATUS[0]}
  drift="$(git status --porcelain src/migrations)"
  if [ -n "$drift" ] || [ "$status" -ne 0 ]; then  → ::error + exit 1  else  ✅  fi
  ```
  Detection is by **`git status --porcelain src/migrations`** (version-independent — does not depend on the CLI's log wording), scoped to `src/migrations` so nothing else in the working tree can confuse it. A fixed name (`ci_drift_check`) labels any file that *would* be written; it's never committed (the job fails first). `${PIPESTATUS[0]}` captures `migrate:create`'s exit code, not `tee`'s.

### Decisions worth a fresh session knowing

- **git-status detection, not log-string grep.** The busbyk snippet in #10978 greps stdout for "Migration created at"; a `git status` check on `src/migrations` is more robust (survives a log-wording change between Payload versions) and also catches a modified snapshot, not just a new file.
- **Placed in `verify`, runs on every PR and push.** It's a correctness gate, so it belongs where lint/typecheck/test already gate — not only on deploy. It needs no DB (see homework), so it costs nothing extra and can't flake on the Postgres service.
- **Ambiguous rename ⇒ fail, by design.** A rename genuinely needs a hand-authored migration (drizzle can't infer intent), so treating the non-zero exit as drift is correct, not a false positive.

### Verification (what was run, and where)

Same standing rule (the sandbox can't run this project's real toolchain — the `payload` CLI needs the Linux esbuild/tsx binary the Windows-mounted `node_modules` doesn't have, and the config transitively imports next-intl → `next/server`; **CI / Filip's machine is the authority**). **In-sandbox:** the ci.yml re-parses as valid YAML with the step correctly ordered after `test:int`; the step's shell script passes `bash -n`; the detection core was simulated in a throwaway git repo across all four cases (no-drift→pass; file-written→fail; non-zero-exit→fail; `PIPESTATUS[0]` captures the right exit code); and the whole design was grounded line-by-line in the installed `payload@3.86` source (above). **On Filip's machine (2026-07-21):** **both directions confirmed** — a clean tree exits with no file (guard passes), and an injected throwaway field makes `migrate:create` write a migration that the guard detects (guard fails), reverted cleanly afterward. **Green in CI** on the normal in-sync tree (the guard logs `✅ No schema drift`).

### Not done in this slice (so a fresh session doesn't assume otherwise)

## Pre-launch SEO + mobile-legibility polish (built + verified in CI 2026-07-21)

The follow-up the audit-gate slice deliberately left open (a **hard launch gate**), finally closed. Chosen as the next slice because it is the one substantial, **fully in-repo, unblocked** piece of launch-readiness left: Phase 4 part 2 (SES email) is gated on the `bulbau.lu` domain + a verified SES/DKIM identity, Phase 5 (translations) is gated on a DeepL key, and Phase 4 part 1's staging render is gated on the manual IAM apply — none of which can be done from code alone. **CSS + metadata + one config file only — no schema, no migration, no infra change** (`git revert`-able).

### The homework (re-checked at implementation time, per the standing rule)

- **`legible-font-size` (Lighthouse SEO):** passes when **≥60% of a page's text (by character count) is ≥12px**; best practice is "never below 12px". So the fix is simply to floor every UI label at 12px.
- **`tap-targets` / target size:** WCAG 2.2 **SC 2.5.8 (Level AA) = 24×24 CSS px** minimum, *or* the spacing exception (a 24px-diameter circle centred on the target doesn't intersect a neighbour). ## Phase 4, part 1 — PDF quote generation + Download (built + verified 2026-07-20)

### The engineering decisions (homework — worth a fresh session knowing)

TECHSPEC §3/§6.5 originally named **"Playwright + `@sparticuz/chromium-min`"**. Re-researched at implementation time (the standing rule that "current best practices" are re-checked, not trusted from when the plan was written), the built reality — and why it diverges — is:

- **`puppeteer-core` + the full `@sparticuz/chromium` (not `-min`), on an x86_64 function.** The npm `@sparticuz/chromium` package **ships x64 binaries only**; arm64 support (since Chromium v135) exists **only** as a Lambda-layer zip or a self-hosted remote *pack tar* that the `-min` package downloads at cold start. Using `-min` + a remote pack would mean versioning and hosting an extra artifact and paying a cold-start download — more moving parts, against the "periodic-maintenance-only" goal. So the PDF function is **x86_64 with the Chromium binary bundled in the function** (self-contained, nothing fetched at runtime). The main `Web` function stays **arm64** — the two are isolated (which is exactly why TECHSPEC §13 wants the split), so the architecture mismatch is irrelevant, and the cost delta on a rarely-invoked pay-per-call function is negligible. Puppeteer vs Playwright: either drives headless Chromium fine for `setContent`→`pdf`; `puppeteer-core` is the pairing `@sparticuz/chromium`'s own docs target, so it was chosen.
- **The PDF function is a *dumb, stateless HTML→PDF renderer* with no Payload and no DB access.** All business logic (pricing via `src/lib/pricing/`, i18n, company data, the §7 contact state) lives in the main app, which assembles finished HTML and passes `{ html }` to the function; the function only launches Chromium and prints. This matches §13's least-privilege intent ("the PDF function has SES but no database access") and keeps the risky Chromium bundle entirely out of the hot-path app function.
- **The route lives at `src/app/api/quote/route.ts`** (outside the `[locale]` segment — locale travels in the POST body), so next-intl's middleware never rewrites it and it isn't touched by the `/admin` proxy (both exclude `/api`). It coexists with Payload's `(payload)/api/[...slug]` catch-all — a static `/api/quote` outranks the catch-all (confirmed by `build`).

## Phase 3, part 1 — live price calculator + shared evaluator (built + verified 2026-07-19)

The first user-visible slice of Phase 3: the public **real-time price calculator**, replacing the Phase 2 static/disabled preview. Deliberately bounded to **the shared pricing evaluator + the public calculator**, leaving the admin **visual Calculator Field/Formula Builder** (the non-technical authoring UI, TECHSPEC §6.4) to **Phase 3 part 2** — the same "prove the risky shared foundation first, then the authoring UI" split used for Phase 2. Rationale: the novel engineering (a safe evaluator shared by client + server, the §7 edge cases, the real-time UX) all lives in the evaluator + public component; once proven against hand-authored `formula` JSON, the visual builder is a self-contained admin component that emits that same JSON. **Migration-free** — the `calculatorFields` array + `formula` JSON have existed since Phase 1.

### The engine decision (homework — worth a fresh session knowing)

TECHSPEC §5/§6.3 named `json-logic-js` "(or equivalent safe evaluator)". Re-researched at implementation time (the standing rule that "current best practices" are re-checked, not trusted from when the plan was written): **`json-logic-js` is effectively unmaintained** (no npm release in 12+ months) — a liability against the "stay-current / periodic-maintenance-only" goal. Its maintained successor is **`json-logic-engine`** (MIT, TS-native, ESM+CJS, ~5× faster, v5.0.7). **Decision: keep JSONLogic as the stored/interchange format** (portable, library-agnostic, exactly what the part-2 visual builder will emit) **but evaluate it with a ~50-line zero-dependency in-house arithmetic-subset interpreter** rather than adding either library. Why: the pricing domain is tiny and closed (`var`, `+ − × ÷`, `min`, `max`); owning the evaluator gives full, unit-tested control over the money-math edge cases that actually matter (division by zero, NaN/Infinity, the §7 non-positive-total rule), adds **zero supply-chain surface** — keeping the strict `npm audit --audit-level=high` CI gate clean (recall the `tmp`/lhci episode, where one added dev tool breached that gate) — and involves **no lock-in**: because the on-disk format is standard JSONLogic, swapping in `json-logic-engine` later (if operator needs ever outgrow arithmetic) is a drop-in with **no data migration**. This is the same "custom because the domain is small, well-understood, and we want control" reasoning used for the TOTP crypto.

### What was built (files in this slice)

- **`src/lib/pricing/jsonlogic.ts` (new)** — the safe interpreter. `evaluateJsonLogic(rule, data)` over a closed operator set (`var` with default; `+ − × ÷`; unary `−`; `min`/`max`; nesting for order-of-operations). Never `eval`/`new Function`; an unknown operator or malformed rule **throws** (callers treat a throw as "can't price it"). `isUsableFormula()` distinguishes a real formula from `{}`/`null`/absent. Exports the `JsonLogic`/`LogicData` types.
- **`src/lib/pricing/index.ts` (new)** — the pure domain wrapper (no React / next-intl / DB, so it runs **identically on the client, on the server, and in the Phase 4 PDF**): `toPricingFields` (projects Payload's `calculatorFields` → a client-safe `PricingField[]` — plain strings, locale-resolved labels), `coerceFieldValue`/`coerceInputs` (number/dropdown/toggle → a numeric scope), `fieldContribution` (signed `unitPrice × value`, or null when a field has no unitPrice), `computePrice` (**default path** = Σ signed contributions; **custom-formula path** = the JSONLogic evaluated over raw field values, authoritative; §7 non-positive/non-finite → `{ kind: 'contact' }`; rounds to 2 dp guarding float error; **never throws**), and `formatCurrency` (locale-aware EUR via `Intl.NumberFormat`).

### Decisions worth a fresh session knowing

- **Two price paths, mutually exclusive for the total.** Default (no formula) = Σ signed `unitPrice × value` — covers the common service with no formula to author, and matches the field-level model in FUNCTIONALITY §5.3. A custom JSONLogic `formula`, when present, is authoritative and evaluated over **raw field values** (fixed costs, %-adjustments, groupings live there). The breakdown shows per-field contributions in the default path (they sum to the total) and entered values under a formula ("where applicable", §4).
- **Required-field gating over always-live.** An "estimate" that silently treats unfilled required inputs as 0 invites false trust; gating on **required only** (not all fields) keeps it responsive (optional fields still default to 0 and update instantly). Hard, blocking validation (format errors, etc.) belongs at PDF-generation time (Phase 4). **0 is a value, blank is not.**
- **The evaluator is pure and framework-free**, so it's the one piece fully unit-testable in-sandbox (Node `--experimental-strip-types`) and reusable verbatim by the Phase 4 PDF and any server-side validation — the on-screen price can never drift from the quoted price (TECHSPEC §3/§6.3).

## Phase 3, part 2 — admin visual Formula Builder + live preview (built + verified 2026-07-20)

The authoring counterpart to part 1: the **non-technical UI that produces** the `formula` part 1 already evaluates. Same "prove the risky shared foundation first, then the authoring UI" split used for Phase 2 and Phase 3 part 1. **Migration-free** — the `calculatorFields` array + `formula` JSON have existed since Phase 1, and this slice changes no schema (only admin-side UI + a pure lib + tests).

### Scoping decision (worth a fresh session knowing)

TECHSPEC §6.4 / FUNCTIONALITY §5.3 describe two things: a **Calculator Field Builder** and a **Formula Builder**. On review, the **field builder is already met by the native Payload `calculatorFields` array** — it gives label / type / options / unit price / +/− sign / required / drag-reorder as first-class field types with zero custom code, which is exactly §5.3's field-builder list. So the genuinely un-built, novel piece is the **Formula Builder + live preview**, and that is what this slice delivers. (Re-implementing the array editor as a custom component would only add risk and duplicate native behaviour.) This is the same "build the risky/novel part, reuse the platform for the rest" judgment used throughout.

### The core design decision: the builder is a *view* over canonical JSONLogic

The stored `formula` value **stays plain JSONLogic** (never the builder's own model). Rationale: the shared evaluator (`src/lib/pricing/`), the public `ServiceCalculator`, and (Phase 4) the PDF all read `formula` as JSONLogic — storing the builder's model there would break all three and force a data migration. So the builder **compiles its structured model → JSONLogic on every edit** (`setValue`) and **parses JSONLogic → model on load**. Anything the builder can produce round-trips losslessly; anything hand-authored outside its canonical shape (e.g. `min`/`max`, division) is detected (`parseFormula` returns `null`) and shown in a **raw-JSON fallback editor** — nothing is ever locked out. This is why the whole compile/parse core is a **pure module** (no React/DB), unit-testable in-sandbox and shared verbatim by the component and the tests.

### What was built (files in this slice)

- **`src/lib/pricing/formulaBuilder.ts` (new, pure)** — the builder model (`BuilderFormula` = ordered `terms[]` + `adjustments[]`) and the two conversions: `compileFormula(model)` → `JsonLogic | null` (null when empty ⇒ the pricing engine's default per-field summation path) and `parseFormula(rule)` → `BuilderFormula | null` (null ⇒ not builder-canonical ⇒ raw fallback). Terms are field (`{var} × ±multiplier`), fixed cost (`±number`), or one-level group (`(Σ members) × factor`, factor a constant or a field, optionally negated). Adjustments are percentages compiled as `× (1 ± p/100)` and peeled back on parse (VAT = `×1.1`, discount = `×0.9`). Emits **only** the operators the evaluator supports (`var`, `+`, `*`); `min`/`max`/`−`/`÷` stay available to raw-JSON authors but aren't surfaced (kept focused on §5.3's requirements).
- **`src/components/admin/FormulaBuilder.tsx` (new, `'use client'`)** — the custom field mounted on `Services.formula` via `admin.components.Field`. Reads the sibling `calculatorFields` live from form state (`useAllFormFields` + `reduceFieldsToValues`) so field dropdowns/preview update as the operator edits fields; edits the value via `useField`; term/group/adjustment editors with add/remove/reorder (up/down, no dnd lib — zero new deps); an **"Edit raw JSON" ⇄ "Use visual builder"** toggle (switching to the builder from a non-canonical rule warns before replacing); and a **live preview** that runs the exact same `computePrice()` the public page uses.
- **`src/components/admin/FormulaBuilder.scss` (new)** — styled with Payload's `--theme-*` variables so it adapts to light/dark automatically.
- **`src/collections/Services.ts` (modified)** — the `formula` field gains `admin.components.Field: '/components/admin/FormulaBuilder#FormulaBuilder'` and a reworded description.
- **`src/app/(payload)/admin/importMap.js` (modified)** — registered the component (Payload regenerates this at startup / `generate:importmap`; the committed entry is kept correct so it's never stale).
- **`tests/int/publicSite.int.spec.ts` (modified)** — 8 new pure tests: empty→null, field/fixed/mixed round-trips, VAT `×1.1` + discount `×0.9`, adjustment-order preservation, group round-trips, non-canonical→null, and one that **evaluates the compiled JSONLogic through the real `evaluateJsonLogic`** to prove preview == production at the arithmetic layer.

### Decisions worth a fresh session knowing

- **Admin UI is English-only** (like the existing TOTP views), so the trilingual next-intl catalogs were untouched — the **109/109/109 key parity holds**. The Payload admin is an owner-facing surface; the public trilingual site is unaffected.
- **Percentage adjustments cover VAT/discount without a new operator** — `× (1 ± p/100)` uses only `*`, so the evaluator's closed operator set was sufficient and **no evaluator change (and no migration) was needed**. Confirmed before building.

This preserves "preview == production" — the whole point of the shared module.

### Issues found during testing (and fixed) — worth a fresh session knowing

1. **Emptied rich-text ≠ null — the disclaimer fallback bug.** Clearing a service's `disclaimer` in the admin doesn't store `null`; Lexical stores an *empty document* (`{ root: { children: [empty paragraph] } }`), a truthy object. 2. **`push` ≠ `migrate` — the local backfill gotcha (not a bug).** After deleting a service locally, a *pre-existing* project lost its label. Cause: local dev runs Drizzle `push` (schema sync on boot) which **does not run migrations**, so the migration's backfill `UPDATE` never populated `serviceName` for rows created before this slice — they stayed `NULL`, and on delete there was nothing to fall back to. New/edited projects are fine (the hook), and **on staging this doesn't happen** because CI runs `payload migrate` (backfill included). To reconcile a local DB: re-save each project once (fires the hook) or run the backfill `UPDATE` by hand. **General lesson:** local `push` never applies migration `up()` bodies — any data backfill in a migration only runs on deployed stages (or via an explicit local `migrate` against a non-prod DB), so pre-existing local rows won't reflect it.
3. **Deleting a route leaves stale generated types.** After deleting `[id]/`, `tsc`/`build` failed on `.next/**/types/validator.ts` still importing the removed `[id]/page.js`. **Fix:** clear `.next` (`npm run devsafe`, or delete the folder) so Next regenerates the route-type validators. Harmless, but confusing if you don't know Next caches per-route type validators.

## Phase 1, part 1 — TOTP 2FA (built 2026-07-06)

Payload has no native 2FA, so this is custom code layered on top of Payload's own email/password auth, per `docs/TECHSPEC.md` §6.6. `docs/FUNCTIONALITY.md` §5.1 treats 2FA as **mandatory, not optional**.

### What was built (files in this slice)

- `src/lib/totp/` — the crypto/OTP toolkit, all Node-runtime:
  - `keys.ts` — one root secret (`TOTP_ENCRYPTION_KEY`, separate from `PAYLOAD_SECRET` for key separation) → two HKDF-derived subkeys (one for AES-256-GCM encryption, one for HMAC signing).
  - `crypto.ts` — AES-256-GCM encrypt/decrypt of the TOTP secret at rest (`iv:authTag:ciphertext`, base64).
  - `otp.ts` — `otplib` wrappers (SHA-1/6-digit/30s defaults, matching mainstream authenticator apps); `epochTolerance: 30` (±1 window) and `afterTimeStep` replay protection.
  - `stepUpToken.ts` — plain HMAC-SHA256 "second-factor-verified" token (`uid`, `exp`), 2h TTL matching Payload's session; `timingSafeEqual` compare.
  - `requestHelpers.ts` — parse/verify the step-up cookie from headers or Payload's cookie map; build set/clear `Set-Cookie` values.
  - `rateLimit.ts` — Upstash Redis sliding-window limiter (5/5min) when `UPSTASH_REDIS_REST_URL/TOKEN` are set; an in-memory fallback for local/CI (same "real service in prod, local stand-in for dev/CI" pattern as S3Mock; not safe multi-instance, warns if it ever activates in prod).
  - `qr.ts` — QR data-URL for the enrollment screen.
- `src/collections/Users.endpoints.ts` — custom endpoints under `/api/users/totp/*`: `setup` (mint secret + QR; re-enroll requires current step-up), `enable` (confirm a code, flip `totpEnabled`, set step-up cookie), `verify` (per-login second factor, sets step-up cookie, user+IP rate-limited), `disable` (requires step-up **and** password re-check), `status`.
- `src/access/requireTotpVerified.ts` — access wrapper: denies unless there's a `req.user`, `totpEnabled`, **and** a valid step-up cookie for that user id; then delegates to the wrapped access fn. **This is the security boundary.**
- `src/proxy.ts` — Next.js 16 Node-runtime request interceptor (the `proxy.ts` convention that replaced `middleware.ts` in Next 16) gating `/admin/:path*`: a password-only session hitting a protected admin route is redirected to `/admin/totp-verify`. UX/defense-in-depth; verifies the `payload-token` JWT itself (see the key-derivation bug below).
- `src/components/admin/` — `TotpSetupView`/`TotpSetupForm`, `TotpVerifyView`/`TotpVerifyForm` (new Payload admin root views at `/admin/totp-setup` and `/admin/totp-verify`), and `BeforeDashboardTotpGate` (a `beforeDashboard` redirect gate). Registered in `payload.config.ts` and `src/app/(payload)/admin/importMap.js` (regenerated, not hand-edited).
- `src/collections/Users.ts` — hidden, non-API-readable `totpSecret`/`totpEnabled`/`totpLastTimeStep` fields; the five endpoints; `requireTotpVerified`-wrapped access; Payload's own `maxLoginAttempts: 5` / `lockTime: 10m` for the password step.
- `src/collections/Media.ts` — writes (`create`/`update`/`delete`) now 2FA-gated; public `read` unchanged (the public site fetches media directly).
- Env/CI/docs: `TOTP_ENCRYPTION_KEY` + Upstash vars added to `.env.example` and CI (`ci.yml`); README gained a 2FA section; `payload.config.ts` wires the views/components; `payload-types.ts` regenerated.
- Tests: `tests/int/totp.int.spec.ts` (crypto round-trip, replay protection, rate-limit fallback, access wrapper, field-hiding), `tests/int/proxy.int.spec.ts` (the proxy's gate logic in isolation), `tests/e2e/totp.e2e.spec.ts` (full enrollment + new-session flows); `tests/helpers/seedUser.ts` gained `seedUnenrolledTestUser`/`seedEnrolledTestUser`; `tests/helpers/login.ts` completes the TOTP step.

### Two real bugs surfaced during testing (both worth a fresh session knowing)

1. **Proxy verified the session JWT against the wrong key — the headline bug.** `src/proxy.ts` verified the `payload-token` with the **raw** `PAYLOAD_SECRET`. Payload does **not** sign with the raw value: on init it derives `crypto.createHash('sha256').update(config.secret).digest('hex').slice(0, 32)` (see `node_modules/payload/dist/index.js`, `BasePayload` init) and signs/verifies with `TextEncoder().encode(...)` of **that**. So every real token failed with `signature verification failed`; `verifiedPayloadUserId` returned `undefined`; the proxy fell through to `NextResponse.next()` and never redirected — the symptom was a password-only session landing on `/admin/collections/users` and seeing an empty "Nothing found" list (the access wrapper correctly denied the data; only the redirect UX was missing). The integration test masked it because it hand-signed its test tokens with the raw secret too — test and buggy proxy agreed with each other while both disagreed with reality. **Fix:** `payloadJwtKey()` in `proxy.ts` reproduces Payload's exact derivation and is **exported and reused by the int test**, so the two can't drift again. **Lesson:** the one piece of custom code re-deriving Payload's own behavior is exactly where reality diverged — from a confidently-worded but unverified comment ("does not hash or transform"). Verify such assumptions against the installed source, and never let a test re-implement the very thing it's meant to check.
   - **Diagnostic red herring worth not repeating:** this was first mis-attributed to Turbopack "not running" the proxy, because `.next/.../middleware-manifest.json` was empty. That file is empty *by design* for a **Node-runtime** proxy — those register in `functions-config-manifest.json`, not `middleware-manifest.json`. A brief switch of `dev` to `--webpack` was a wrong turn (it only added slow-compile timeouts) and was reverted; `dev`/`devsafe` stay on Turbopack. What actually pinned it was temporary `console.warn` instrumentation in the proxy: the dev-server log showed `jwtVerify FAILED: signature verification failed`, which is unambiguous. When a Node-runtime proxy "doesn't fire," instrument it and read the server console rather than reasoning about manifests.
2. **TOTP replay-window flake in the e2e suite (a test bug, not a code bug).** The "new session" test reused the account the enrollment test had *just* enrolled. When its login code fell in the same 30-second TOTP window whose time-step enrollment had already consumed, otplib's replay protection (correctly) rejected it — a ~50–70%-of-runs timeout depending on where the window boundary fell. The 2FA logic is right; the test was unrealistic (real "new session" logins happen long after enrollment). **Fix:** `seedEnrolledTestUser` seeds a dedicated, already-enrolled account with an untouched time-step; the new-session test uses that, which also removes a hidden test-1→test-2 ordering dependency.

### Verification
- `npm run test:int`: green (both new int files; `proxy.int.spec.ts` now signs with the derived key via the shared `payloadJwtKey`).
- `npm run test:e2e`: **now green in CI.** The `deploy-staging` job that shipped Phase 1 `needs: verify`, which runs the full e2e suite — so all 6 passed there (the replay-flake fix included), plus Filip's earlier manual redirect check. `npm run devsafe` (clean `.next` + restart) is required after any `proxy.ts` change — Next never hot-reloads the proxy file.

### Decisions worth a fresh session knowing

- **Push in dev/CI, migrate on deployed stages** — Payload's own recommended split, kept (not "push:false everywhere"). CI's `verify` job keeps pushing on boot against its ephemeral Postgres, so the test suite is untouched; only real stages are migrations-only. Having committed migrations present does **not** affect push mode (push ignores the migration files), so `test:int`/`test:e2e` stay green.
- **Migrate in CI, not `prodMigrations`** — the serverless-correct choice (see above), and what Payload's docs recommend for anything that isn't a long-running container.
- **Neon: migrations use the DIRECT (unpooled) URL** — DDL breaks through Neon's PgBouncer pooler (transaction pooling). Runtime stays pooled; only the CI migrate step uses the direct URL. This is the single most likely thing to get wrong, so it's called out in the workflow comment, README, `.env.example`, and TECHSPEC §10.5.
- **`TotpEncryptionKey` required, Upstash optional** — encodes the actual severity: no TOTP key = broken admin (hard-fail the deploy); no Upstash = degraded-but-working rate limiting (tolerated pre-launch).

### Sandbox constraints hit (and how they were worked around)

- **The `payload` CLI can't load the TS config in-sandbox** — it uses `tsx`→esbuild, and `@esbuild/linux-x64` isn't in the Windows-mounted `node_modules` (the same "native binaries are Windows-only" quirk from Phase 0). Worked around exactly as the content-model session did: load the config with Node 22's native `--experimental-strip-types` + a small `@/`-alias/extensionless resolve hook, and call `createMigration`/`migrate` on `payload.db` directly. One wrinkle: Payload's generated migration imports `MigrateUpArgs`/`MigrateDownArgs` as *values* (`import { MigrateUpArgs, MigrateDownArgs, sql }`), which `--strip-types` can't elide (only real `tsx`/esbuild does), so the sandbox apply-test patched that one import line in a scratch copy; **the committed file keeps Payload's original import** (correct for CI/Filip's toolchain).
- **`bwrap --unshare-pid --die-with-parent`** — every bash call is its own PID namespace, so **no process survives between calls** (background/`setsid`/`nohup` all die with the call). But the on-disk `pgdata` persists, so the pattern is: warm the cluster (`initdb`) once, then reuse the data dir in later calls (start is ~0.4s, no initdb). Also note `pkill -f postgres` **self-matches** the calling shell (its command line contains "postgres") and kills the call — use `pkill -x postgres` or, since nothing survives a call anyway, don't bother.
- **Stale mount reads** — after editing `package.json` via the Edit tool, the Linux side served a truncated pre-edit copy (2473 bytes, unterminated mid-string), which broke Node's module resolution for any in-sandbox eval. The Read tool (authoritative Windows view) confirmed the file was correct; the adapter-options check was rerun mount-independently (importing `@payloadcms/db-postgres` by absolute path). Same Windows↔Linux flakiness as always — trust Read/Edit/Write.

## How Phase 0 went (the Lambda spike)

Scaffolding (Next.js 16.2 + Payload 3.85 + Postgres/Drizzle + SST v4/OpenNext + GitHub Actions) went roughly as planned; getting Payload to actually serve on Lambda took five real, non-obvious fixes, none of which invalidated the architecture:
1. OpenNext copies externalized packages (`pg`, `pino`) into the Lambda bundle via symlinks that Windows can't reliably produce as Linux symlinks → **deploys go through CI (Ubuntu) only, never native Windows.**
2. `sharp` is excluded from the main server bundle by OpenNext and ships arch-specific binaries that don't cross-compile (x64 CI → arm64 Lambda) → omitted entirely (nothing in Media uses it; it's now also removed from `package.json`, keeping deps honest).
3. Media uploads need an explicit storage adapter (`@payloadcms/storage-s3`) — SST linking a bucket only wires IAM, it doesn't make Payload use it.
4. `admin/importMap.js` is generated, not static — regenerate (`npm run generate:importmap`) after any collection/field/plugin that contributes admin UI; silently stale otherwise.
5. IAM's `aws:RequestedRegion` doesn't scope global services (CloudFront/IAM/Route 53) → CloudFront needed its own unconditioned statement.

## Environment quirks worth knowing (specific to this Cowork/Claude sandbox, not the app)

- The Linux sandbox's view of the Windows-mounted repo has intermittent caching problems: `git status`/`git diff`/`cat` sometimes show stale or truncated content. **Trust the Windows-native `Read`/`Edit`/`Write` tools over bash; cross-verify with `git show HEAD:<path>`.** When bash's copy of a file looks stale, rewrite it via a heredoc (`cat > path <<'EOF' ... EOF`) from content confirmed by `Read`.
- **Do not run `git add`/`commit`/`push` from the sandbox** — a stale `.git/index.lock` once could not be removed (`rm` returns `Operation not permitted` on the Windows mount). Filip commits via his own tooling (GitHub Desktop). The same permission quirk means the sandbox **cannot delete files** on the mount — e.g. `smoke-test-tmp.mjs` (a harmless leftover, untracked) had to be left for Filip to delete manually.
- `npm install`/`sst install`/large builds can exceed a single sandbox command's time limit; retrying converges.
- **Turbopack cannot run in this sandbox** — only WebAssembly SWC bindings are available, and Turbopack requires native bindings (`Turbopack is not supported on this platform`). For any in-sandbox `next build`/`next dev` reproduction, use `--webpack`. Filip's real machine (Windows) uses Turbopack, which is the Next 16 default for both `dev` and `build`; keep that distinction in mind when reproducing bundler-adjacent behavior (it's why the Turbopack red herring above was hard to rule out from the sandbox alone).
