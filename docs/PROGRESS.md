## Media upload/delete test coverage — closed (2026-07-04, same-day follow-up to Phase 0)

Phase 0 signed off with one flagged gap: media upload/delete through S3 was only ever manually verified in a browser, never covered by an automated test. Both `docs/PROGRESS.md` (this file, previous version) and `docs/TECHSPEC.md` §13 called this out as the thing to close before Phase 1 added more upload-dependent fields (hero images, project photos) on top of untested ground. This section is the record of doing that.

### What the docs assumed vs. what was actually current

The prior version of this file proposed closing the gap with "either real AWS credentials in the CI `verify` job or a MinIO/LocalStack service container." Current research (done before writing any code, per the project's standing workflow) found neither MinIO nor LocalStack still fit that description by the time this was picked up:

- **MinIO** stopped publishing free Community Edition Docker images on Docker Hub/Quay on 2025-10-23 (announced abruptly, during a security-patch release, which drew real community backlash), and progressively stripped admin-console features from the Community Edition through 2025 (site replication, security, encryption flagged as next). By 2026-02-12 the repo was marked "no longer maintained," and formally archived (read-only) by 2026-04-25.
- **LocalStack** ended its Community Edition on 2026-03-23. Core AWS services including S3 moved behind a paid tier as of LocalStack 3.0; the free "Hobby" plan that remains is explicitly non-commercial-use only — a real constraint for bulbau.lu, which is a commercial site.

Neither is a reasonable pick anymore for a small commercial project's CI pipeline. Used **Adobe's S3Mock** instead: Apache-2.0, actively released (5.1.0 shipped the month before this was written), purpose-built specifically for this exact scenario (mock S3 for integration testing) rather than being a general AWS-emulation product with a business model to protect — which is exactly the kind of incentive that produced the MinIO/LocalStack situation in the first place. This is the kind of divergence-from-the-written-plan the project's standing instructions call for surfacing and acting on without waiting for sign-off first; the docs are updated now to match.

### What was built

- `docker-compose.yml` / `.github/workflows/ci.yml`: an `s3mock` service alongside the existing ephemeral Postgres (`adobe/s3mock:5.1.0`, auto-creating a bucket on startup via `COM_ADOBE_TESTING_S3MOCK_STORE_INITIAL_BUCKETS`).
- `src/payload.config.ts`: the `s3Storage()` plugin config now branches on whether `S3_ENDPOINT` is set — unset (real AWS, staging/production or a real bucket in local dev) behaves exactly as before; set (S3Mock) adds `forcePathStyle: true` and dummy credentials, and — after a bug found via manual testing (below) — also turns `clientUploads` off.
- `tests/int/media.int.spec.ts`: creates a Media doc via the Local API with a real file buffer, confirms the object actually lands in S3 via a direct `HeadObjectCommand` against a separately-constructed S3 client (not just trusting Payload's own DB row), then deletes the doc and confirms the object is actually gone the same way.
- `.env.example` / `README.md`: fixed a pre-existing, unrelated doc/reality gap found in passing — `README.md` told developers to set `S3_BUCKET`, but `.env.example` never mentioned it at all. Both now document the S3Mock path (no real AWS credentials needed for local dev) and the real-bucket path side by side.
- `.github/dependabot.yml`: added a `docker` ecosystem entry so the pinned `adobe/s3mock` tag in `docker-compose.yml` gets update PRs. Note: this does **not** cover the image tags pinned inside `.github/workflows/ci.yml`'s `services:` block (Dependabot's docker ecosystem doesn't parse GitHub Actions workflow files) — those, and the pre-existing `postgres:17-alpine` pin, remain a manual periodic-review item, not a new gap this introduced.
- `package.json`: added `@aws-sdk/client-s3` as an explicit devDependency (it was already present transitively via `@payloadcms/storage-s3`, but relying on hoisting for a direct import is fragile).

### Three real bugs this surfaced (not just the intended coverage gap)

Writing this test found genuine, previously-latent bugs — not flakiness, not test-writing noise. Each is worth a fresh session understanding, since the same class of mistake could resurface if e.g. more int test files are added later, or the S3 config is touched again:

1. **Vitest file-parallelism race against shared Postgres.** Every int test file calls `getPayload(config)` in its own `beforeAll`, which triggers Payload's dev-mode schema push (`drizzle-kit push`) against the ephemeral Postgres service container. With only one int test file (`api.int.spec.ts`) this was invisible; adding a second (`media.int.spec.ts`) meant Vitest's default file-level parallelism ran both schema pushes concurrently, and `drizzle-kit push` isn't safe under concurrent execution (check-then-create, no locking) — the second push lost the race with `relation "payload_preferences" already exists`, first seen in a CI run. Fixed in `vitest.config.mts` with `fileParallelism: false` — correct, not a workaround: these tests share one stateful Postgres instance, so treating them as unable to run in parallel is simply accurate.
2. **Config drift between the test's own S3 client and `payload.config.ts`'s.** The test verifies uploads with its own separately-constructed `S3Client` (deliberately — so it can't pass on DB state alone while the real object is missing). That client initially hardcoded `forcePathStyle: true` and dummy credentials unconditionally, instead of mirroring `payload.config.ts`'s "only override when `S3_ENDPOINT` is set" branch. Ran fine in CI (which always sets `S3_ENDPOINT`) but broke the moment Filip tested locally against a real bucket (no `S3_ENDPOINT`): Payload's own upload succeeded (correct behavior, real credential chain), but the test's verification client forced path-style + fake credentials onto a real-AWS request, which the SDK couldn't turn into a readable error — surfaced only as an opaque `Unknown: UnknownError`. Fixed by giving the test client the exact same conditional as `payload.config.ts`. Lesson: any code that talks to the same external service as the app under test needs to share config logic, not reimplement it, or the two will silently diverge.
3. **S3Mock doesn't reliably support CORS for browser-based uploads.** `clientUploads: true` (needed for real AWS, to avoid routing file bytes through the Lambda's payload-size limit) makes the admin UI's browser upload a genuine cross-origin request — browser directly PUTs to the S3 endpoint via a presigned URL. S3Mock's CORS support has a multi-year history of being incomplete ([adobe/S3Mock#74](https://github.com/adobe/S3Mock/issues/74), never fully resolved for uploads specifically), so that PUT's preflight gets blocked, and the browser reports nothing more specific than `TypeError: Failed to fetch` — this was **only caught by manual browser testing**; the Local-API-based automated test never exercises the `clientUploads` code path at all (by design — see point 2's design rationale), so it couldn't have caught this. Fixed with `clientUploads: !process.env.S3_ENDPOINT` — off against S3Mock (browser posts to the same-origin Next dev server instead, which then relays to S3Mock server-side, a call browser CORS rules don't apply to at all), on for real AWS. There's no Lambda payload-size constraint in local dev to trade off against, so this is a strict improvement locally, not a compromise. **This is also why the manual admin-UI pass in the testing guide mattered and wasn't redundant with the automated test** — they cover genuinely different code paths (Local API vs. REST/browser), and only one of the two ever exercises `clientUploads`.

A fourth issue turned out to be pure local misconfiguration, not a code bug: partway through, `test:int` failed locally with the same `UnknownError` even after bug #2's fix, traced to Filip's own `.env` having `S3_BUCKET` set (to a bucket that, confusingly, shares a name — `bulbau-dev` — with the one S3Mock auto-creates) but not `S3_ENDPOINT`/`S3_ACCESS_KEY_ID`/`S3_SECRET_ACCESS_KEY`, meaning the app was actually talking to a real AWS bucket the whole time using whatever ambient AWS credentials happened to be on that machine. Fixed directly in `.env` (gitignored, local-only, no repo change). Worth remembering: **a `SignatureDoesNotMatch` or opaque S3 SDK error during local dev is worth checking `.env` for a missing `S3_ENDPOINT` before assuming the code is wrong** — real AWS and S3Mock produce very different failure shapes for the same underlying mistake.

## How Phase 0 actually went

The scaffolding part (Next.js 16.2 + Payload 3.85 + Postgres via Drizzle adapter + SST v4/OpenNext + GitHub Actions CI/CD) went roughly as planned. The Lambda spike did not — getting a real deploy to actually serve traffic correctly took five rounds of "deploy, hit a real bug, fix it, redeploy," each a genuine, non-obvious issue rather than a typo or a misconfiguration:

1. **`pg`/`pino` "Cannot find module" at runtime, despite a clean build.** OpenNext's build tracing treats certain packages as "external" — instead of bundling them inline, it copies the real files into the Lambda package via a symlink step. On Windows, that step either throws `EPERM` outright, or (worse) silently "succeeds" while producing a Windows-native symlink that doesn't survive being packaged for a Linux Lambda runtime. First symptom was a hard `EPERM` crash during local `sst deploy` (fixed by enabling Windows Developer Mode, which grants symlink-creation permission) — but the *deploy that then "succeeded"* shipped a broken package anyway, because the underlying symlink-format mismatch wasn't actually about permissions. **Real fix: never run `sst deploy` from native Windows. Deploys go through CI (Ubuntu) exclusively**, which doesn't have this problem at all.
2. **`sharp` "Cannot find package"**, a different flavor of the same class of issue, plus a second problem layered under it: OpenNext excludes `sharp` from the main bundle by default (it assumes `sharp` is only used by Next's own separate image-optimizer function), and even if forced back in, `sharp` ships architecture-specific binaries that wouldn't match anyway (CI builds on x64, the Lambda is configured for arm64). Since Payload's `sharp` integration is optional as of 3.x and the Media collection doesn't use any sharp-dependent feature (no `imageSizes`/resize options), it was removed from `payload.config.ts` rather than fought. Revisit if/when Media actually needs resizing — options then are matching CI/Lambda architecture, or a prebuilt arm64 Sharp Lambda Layer.
3. **Media upload failing with `ENOENT ... mkdir 'media'`.** Payload's `upload: true` shorthand defaults to local-disk storage, which cannot work on Lambda (read-only filesystem outside `/tmp`). SST provisioning an S3 bucket and `link`-ing it to the Next.js component only wires IAM permissions — it doesn't tell Payload to use it. Fixed by adding `@payloadcms/storage-s3` with `clientUploads: true` (uploads go browser → S3 directly via a presigned URL, deliberately chosen over the server-proxied variant since routing file bytes through the Lambda would hit its payload-size limits on any normal-sized photo — a near-term certainty, not a hypothetical). **Update (same-day follow-up slice, above): `clientUploads` is now conditional — off specifically when testing locally against S3Mock, due to S3Mock's incomplete CORS support. Still on for real AWS, so this original reasoning still holds there.**
4. **`getFromImportMap: PayloadComponent not found`**, twice, for two different reasons (once earlier in Phase 0 when a Media field was simplified, once when the S3 plugin was added) — `admin/importMap.js` is generated, not static, and goes stale silently whenever the admin UI's component set changes. No build error, just a broken admin page at runtime and, in CI, a Playwright timeout waiting for a login field that never rendered correctly. **Standing rule now: run `npm run generate:importmap` after touching any collection, field, or plugin that contributes admin UI**, and never hand-edit that file. **Caveat added this session**: this rule has an exception. `admin/importMap.js` currently registers `S3ClientUploadHandler` unconditionally, which covers both the `clientUploads: true` and `clientUploads: false` branches (an unused registered component is harmless; a missing one breaks the admin page). Regenerating the import map while `S3_ENDPOINT` is set locally (`clientUploads` evaluating to `false` in that process) risks the generator omitting that entry, which would then break the real-AWS branch until regenerated again in that context. Don't regenerate reflexively for `clientUploads` changes specifically — the current committed file already covers both branches.
5. **`cloudfront:CreateInvalidation` AccessDenied** on the GitHub Actions deploy role, despite a broad region-scoped Allow statement (`aws:RequestedRegion: eu-central-1`) intended to cover exactly this kind of thing. CloudFront (like IAM and Route 53) is a global service whose control-plane calls don't satisfy a regional condition — a gap the policy's own design notes had flagged as a security consideration (for the explicit-Deny backstop) without connecting it to the fact that it would *also* block legitimate global-service calls SST itself needs. Fixed with a dedicated, unconditioned `cloudfront:*` statement. Worth remembering if a custom domain gets added later (Route 53 + ACM): expect the identical gap, same fix pattern.

None of this invalidates the Phase A architecture — every fix was either a build/packaging detail (#1, #2) or a missing piece of glue that any Payload-on-Lambda deployment would need regardless of tooling (#3, #4, #5). The spike's actual question — does Payload run correctly on Lambda — is answered yes.

## Decisions made along the way (also reflected in TECHSPEC.md itself)

- **Repo structure**: single app at repo root (`src/`, `package.json`, `sst.config.ts` all at top level), not a monorepo/workspace split. One deployable unit doesn't need `apps/`/`packages/` — that pattern solves a different problem (multiple apps sharing code). `infra/aws/` was added for the one standalone piece of AWS bootstrap that has to exist before SST/CI can even run (the OIDC trust relationship) — deliberately not folded into `sst.config.ts`'s Pulumi-managed resource graph, since it's the thing that makes that graph reachable in the first place.
- **`frontend/` → `prototype/` rename**: the old top-level `frontend/` folder (holding only the static design-reference prototype) collided in name with the real `src/app/(frontend)/` route group. Renamed, zero references anywhere needed updating (verified via grep before renaming).
- **GitHub Actions ↔ AWS**: OIDC federation (short-lived assumed role), not stored access keys — Filip's explicit preference going in, also the current documented best practice from both AWS's and GitHub's own security docs. Trust policy is scoped to the exact repo *and* GitHub Environment (`environment: staging`), which changes the shape of the OIDC subject claim GitHub issues — using a plain branch-ref condition instead would have been wrong for a workflow that sets `environment:`.
- **IAM permission policy for the deploy role**: broad-but-region-scoped Allow, backstopped by an explicit Deny on the highest-risk IAM/org/billing actions, rather than a hand-enumerated least-privilege policy. This mirrors SST's own documented guidance (hand-enumerating exact actions per component is called out, in SST's own docs, as tedious and error-prone) and its recommended path: start reasonably scoped, then run IAM Access Analyzer against real CloudTrail activity to generate an actually-precise policy once there's usage history to analyze. **That tightening pass is a tracked follow-up, not done yet** — don't reuse this exact policy for a `production` role without doing it first.
- **Cost model**: re-verified against `eu-central-1` pricing at sign-off (see `docs/TECHSPEC.md` §9). Lambda's perpetual free tier is confirmed still current. Total is realistically €3–8/month at this traffic level, not a single precise figure — false precision isn't worth claiming here.
- **S3Mock over MinIO/LocalStack for S3 integration testing** (same-day follow-up slice, 2026-07-04): see the dedicated section above. Both alternatives named in the original plan stopped being viable (license/distribution changes, not a technical shortcoming) between when the plan was written and when this was picked up — a reminder that "current best practices" research needs to happen at implementation time, not trusted from whenever a doc was last written, even a few weeks prior.

## Environment quirks worth knowing (specific to this Cowork/Claude working setup, not the app itself)

- The Linux sandbox's mounted view of the Windows-side repo folder has intermittent caching problems: `git status`/`git diff` run via the sandbox's bash sometimes show files as modified/truncated when they are not — confirmed this happened at least once purely from stale cache, not real changes (cross-checked `git show HEAD:<path>` against the Windows-native file read and they matched exactly). **When bash's view of a file looks suspicious, trust `Read`/`Edit`/`Write` (Windows-native) over bash, and cross-verify with `git show HEAD:<path>` rather than trusting `git status`/`git diff` at face value.**
- Similarly, ad hoc JSON/file validity checks run through that same bash session have come back with false failures (e.g. `python3 -c "json.load(...)"` reporting an "unterminated string" for a file that `Read` shows is complete and well-formed) — same root cause, same fix.
- `npm install`/`sst install`/large builds can exceed a single sandbox command's time limit; retrying converges (npm's cache and partial progress persist across calls even though the sandbox itself resets between some turns).
- **New this session — the staleness above can be bad enough that re-reading doesn't fix it.** Several files (`package.json`, `docker-compose.yml`, `src/payload.config.ts`, `.github/workflows/ci.yml`, `README.md`, `.env.example`, `.github/dependabot.yml`) intermittently showed truncated/stale content via bash `cat` even after the Windows-native `Edit`/`Write` tools confirmed the write succeeded, and re-reading via bash repeatedly did not converge on its own. **Working fix: rewrite the file directly inside the bash sandbox via a heredoc (`cat > path <<'EOF' ... EOF`), using the content already confirmed correct via the Windows-native `Read` tool, then re-verify.** This forced the sandbox's copy to match reality and made subsequent `npm run lint`/`npm run typecheck`/`vitest run` reflect the real code instead of a stale snapshot.
- **New this session — git operations from the sandbox can get stuck.** A `git status` call left a stale `.git/index.lock` (0 bytes) that could not be removed from the sandbox (`rm` returned `Operation not permitted` — a permissions quirk on the Windows-mounted drive, not a real repo lock). **Don't run `git add`/`git commit`/`git push` from this sandbox** — verify changes are correct via the Windows-native `Read` tool and the heredoc-resync trick above, then have the repo owner commit via their own tooling (GitHub Desktop, in this project's case) and push. This session's commits were all made by Filip directly, not by Claude, for exactly this reason.
