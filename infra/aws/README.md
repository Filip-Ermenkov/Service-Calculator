# Staging deploy role (GitHub Actions OIDC)

One-time AWS bootstrap so `.github/workflows/ci.yml`'s `deploy-staging` job can run `sst deploy --stage staging` without stored AWS keys. Not managed by `sst.config.ts` or Terraform — this has to exist *before* SST/CI can run, so it's applied once by hand via the AWS CLI. See the root README's Deploying section for the full command sequence.

## Files

- `github-actions-trust-policy.json` — trust policy for the IAM role. Scoped with `token.actions.githubusercontent.com:sub` to `repo:Filip-Ermenkov/Service-Calculator:environment:staging`. Environment-scoped rather than branch-scoped because the `deploy-staging` job sets `environment: staging`, which changes the shape of GitHub's OIDC `sub` claim (see [GitHub's OIDC-in-AWS docs](https://docs.github.com/en/actions/how-tos/secure-your-work/security-harden-deployments/oidc-in-aws)). Replace `ACCOUNT_ID` with the real 12-digit AWS account ID before use.
- `github-actions-deploy-policy.json` — permission policy attached to that role. The bootstrap (`ManageBootstrap*`) and `ManageSecrets` statements are copied from [SST's own documented IAM policy](https://sst.dev/docs/iam-credentials) — those are genuinely scoped to `sst-*`-prefixed resources. The `DeploymentsScopedToRegion` statement is deliberately broad (SST's own docs say hand-enumerating every action their components might need is tedious and error-prone) but constrained to `eu-central-1` via `aws:RequestedRegion`, and backstopped by an explicit `Deny` on the highest-risk IAM/org/billing actions — a region condition alone doesn't meaningfully limit IAM, CloudFront, or Route 53, since those are global services. Replace `ACCOUNT_ID` here too.
- `ManageCloudFront` exists as its own unconditioned statement because of that same global-service gap: CloudFront's control-plane calls don't satisfy `aws:RequestedRegion: eu-central-1`, so `DeploymentsScopedToRegion` silently doesn't cover them — first hit as a `cloudfront:CreateInvalidation` AccessDenied on the first real `deploy-staging` CI run. If a custom domain is added later (Route 53 + ACM), expect the same gap for those and add an equivalent unconditioned statement rather than trying to stretch the region condition to cover them.
- `ManageAppIamRoles` is the IAM instance of that exact same global-service gap. IAM calls carry no region, so `DeploymentsScopedToRegion` never covers `iam:CreateRole` and friends. It went unnoticed until Phase 4 because SST creates one execution role per component (`bulbau-lu-<stage>-*`) and the app's original roles (Web server, image-optimizer, revalidator, warmer) were all created during the initial hand-run bootstrap deploy under admin credentials — CI's restricted role never had to create a role until the new `Pdf` function became the first one it provisioned itself (`iam:CreateRole ... PdfRole ... AccessDenied`). The statement is scoped to this app+stage's own role-name prefix (`role/bulbau-lu-staging-*`) — least-privilege, not `*` — and the high-risk IAM verbs (`iam:CreateUser`, `iam:UpdateAssumeRolePolicy`, `iam:AttachUserPolicy`, …) remain blocked by `ExplicitDenyHighRiskActions`, which an explicit `Deny` always wins. A new role is created with its trust policy in one `CreateRole` call, so `UpdateAssumeRolePolicy` staying denied does not block it. **This policy is applied to the live role by hand (see below) — editing this file alone does not change AWS; the `deploy-staging` job keeps failing until the updated policy is pushed to the role.**

## Before this is "done"

Per SST's own "Minimize permissions" guidance: once staging has run for a few weeks, run IAM Access Analyzer against its CloudTrail activity and replace `DeploymentsScopedToRegion` with the generated least-privilege policy before this pattern is reused for a `production` role. **Status (2026-07-21): staging is live and has run real migrated deploys plus the Phase 2 public-site deploy and the Phase 3/Phase 4 deploys — including the `Pdf` function's own execution role (the first role CI's restricted deploy role provisioned itself, exercising the new `ManageAppIamRoles` statement).** So there's now a good, representative stretch of CloudTrail history — the generated least-privilege policy will now also capture the IAM role-lifecycle actions, not just the region-scoped app actions. This is an active follow-up tracked in `docs/PROGRESS.md` → "Immediate next steps" — not a blocker for current work. Two things that add **nothing** to this policy's surface: the CI database-migration step (it connects to Neon directly, no AWS role), and Phase 2's two new SST secrets (`SiteUrl`/`AllowIndexing`) — SST secrets live in SSM Parameter Store under the `sst-*` prefix already covered by the `ManageSecrets` statement.
